# Хранение данных и автоочистка БД

На сервере включена ежедневная автоочистка, чтобы потоковая телеметрия и треки не раздували SQLite. Справочники и настройки не чистятся автоматически.

## Расписание

- `03:00` по `Asia/Novosibirsk` — сброс видимого трека на главной странице. Это не физическое удаление всей истории: сервер сохраняет отметку `clearSince`, и главная начинает показывать только новые точки.
- `03:20` по `Asia/Novosibirsk` — физическая retention-очистка старых данных из БД.

Таймзона берётся из `DATA_RETENTION_TIMEZONE`, затем `TELEMETRY_TIMEZONE`, затем `APP_TIMEZONE`; если ничего не задано, используется `Asia/Novosibirsk`.

## Сроки хранения по умолчанию

| Данные | Таблицы | Сколько хранится | Что удаляется |
| --- | --- | ---: | --- |
| Подробная сырая телеметрия и треки | `Telemetry`, `RtkTelemetry` | 14 дней | Точки host-телеметрии, веса, координат и RTK-треки погрузчика, включая треки внутри замесов |
| История замесов | `Batch`, `BatchIngredient`, связанные `Violation` | 45 дней после завершения замеса | Завершённые замесы, фактические ингредиенты, нарушения этих замесов |
| События устройств | `DeviceEvent` | 90 дней | Старые события от устройств |
| Закрытые технические предупреждения | `TechnicalWarning` со статусом `RESOLVED` | 90 дней | Только уже закрытые технические предупреждения |
| Закрытые нарушения без замеса | `Violation` без `batchId`, статусы `CLOSED`/`RESOLVED` | 90 дней | Старые отвязанные закрытые нарушения |
| Справочники и настройки | `User`, `Ration`, `RationIngredient`, `LivestockGroup`, `StorageZone`, `TelemetrySettings`, `DigestSettings`, `AppState` | бессрочно | Не удаляются автоочисткой |

Активные замесы, у которых `endTime = null`, автоочистка не удаляет.

## Настройка через переменные окружения

- `DATA_RETENTION_ENABLED=false` — полностью отключить физическую retention-очистку.
- `DATA_RETENTION_RUN_TIME=HH:mm` — время ежедневной физической очистки, по умолчанию `03:20`.
- `DATA_RETENTION_RAW_TELEMETRY_DAYS=14` — срок хранения `Telemetry` и `RtkTelemetry`.
- `DATA_RETENTION_BATCH_DAYS=45` — срок хранения завершённых замесов.
- `DATA_RETENTION_DEVICE_EVENT_DAYS=90` — срок хранения событий устройств.
- `DATA_RETENTION_RESOLVED_TECHNICAL_WARNING_DAYS=90` — срок хранения закрытых технических предупреждений.
- `DATA_RETENTION_ORPHAN_VIOLATION_DAYS=90` — срок хранения закрытых нарушений без замеса.

## SQLite и место на диске

После `DELETE` SQLite обычно освобождает страницы внутри файла БД для будущих записей, но сам файл на диске может сразу не уменьшиться. Retention-задача по умолчанию делает `PRAGMA optimize` и `PRAGMA wal_checkpoint(TRUNCATE)`.

Если нужно именно вернуть место операционной системе, можно включить ночной `VACUUM`:

```env
DATA_RETENTION_SQLITE_VACUUM=true
```

`VACUUM` тяжелее обычной очистки, поэтому он выключен по умолчанию.
